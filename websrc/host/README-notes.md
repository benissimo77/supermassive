Some notes/analysis on the current dashboard quiz editor and recommendations for how to improve in future...

Today was a perfect example of what happens when a system requires you to touch five different interconnected files just to make one tiny change (like making sure an image URL is validated in an ordering question).

Right now, adding one new question type or augmenting an existing one requires paying a massive "architecture tax":

The Backend Schema: You have to surgically edit a massive 500-line quiz-schema.json file with complex allOf/if-then blocks.
The Backend Service: You have to add custom JS validation loops in quizService.js.
The Frontend UI: You have to write raw HTML strings in quiz-editor-registry.js.
The Frontend Serializer: You have to write custom DOM query logic to pull the data out (registry.serialize()).
The Frontend Deserializer: You have to write custom DOM population logic to put the data back in (registry.deserialize()).
The Frontend Error Map: You have to update the hardcoded switch statement in dashboard-quizedit-v2.js to translate the AJV error into plain English.
That is exhausting and prone to human (and AI) error.

The Future-Proofing Strategy (Without Over-engineering)
To stick strictly to our "Anti-Overengineering Guardrails" (Tools over Frameworks, Data is the Common Language), I've documented a persistent memory note (/memories/quiz_editor_architecture.md) so that we never lose sight of this pain point.

When you're ready to tackle this (whether tomorrow or six months from now), here is the exact roadmap to make the editor 100% declarative and bulletproof:

1. Auto-Serialization (The "No More DOM Queries" Tool)
Instead of writing a custom serialize and deserialize function for every single question type, we build a single, shared utility form-binder. You would just write your HTML template with a data-bind attribute:

A single 20-line utility function sitting in dashboard-quizedit-v2.js would automatically scan the DOM and build the JSON payload, and do the reverse when loading the quiz. You immediately delete about 400 lines of tedious DOM querying across the whole registry.

2. Schema Co-location (The "One File to Rule Them All" Tool)
Instead of maintaining a massive centralized quiz-schema.json, each question type in the registry should export its own JSON schema requirements alongside its HTML template. The backend simply imports the registry and automatically generates the complete validation schema on boot. If you delete a question type from the registry, its validation rules vanish simultaneously.

3. Data-Driven Error Mapping
Instead of a giant switch statement in improveValidationErrorMessage that we have to manually update constantly, we handle errors generically by letting the question types define their own readable error names for their specific fields inside their registry entry.

When you come back to the editor, we can implement Step 1 (Auto-Serialization). It wouldn't require any external libraries, just a clever 20-line recursive function that reads data-bind, and it will immediately make adding a new question property as simple as copy-pasting an HTML <input> tag! <input data-bind="newProperty">.
